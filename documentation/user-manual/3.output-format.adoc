Two formats are supported: JSON and Protocol Buffers (experimental).

=== JSON format

JSON is fast and typically smaller in generation overhead.
The writer constructs messages directly during parsing to minimize allocations.

==== Response fields

- scn/scns — SCN value (decimal in `scn` or hex string in `scns`).
See xref:../json/format.adoc#scn[scn].
- tm/tms — timestamp (numeric or string).
See xref:../json/format.adoc#timestamp[timestamp].
- xid/xidn — transaction ID formats.
See xref:../json/format.adoc#xid[xid].
- db — database name.
See xref:../json/format.adoc#db[db].
- payload.op — operation type: `"begin"`, `"commit"`, `"c"`, `"u"`, `"d"`, `"ddl"`, `"chkpt"`.
- payload.schema — included in DML messages; contains `owner`, `table`, `obj`, and `columns`.
- payload.before / payload.after — column value arrays for UPDATE/DELETE and INSERT/UPDATE respectively.
Only values present in redo are emitted; unchanged columns may be omitted.
- payload.rid — row ID when available.
- DDL messages may include the SQL text and parsed system-table DML used to update the internal schema.

[CAUTION]
====
Only data available in the redo log is emitted.
For updates, unchanged columns may be absent.
====

=== Protocol Buffers

Protobuf uses the same logical field names and types as JSON, but constructs messages field-by-field before serialization.
Memory and CPU overhead are higher than JSON.

[IMPORANT]
====
The status of Protocol Buffers format is in Experimental state.
It be unstable and not fully implemented.
====
