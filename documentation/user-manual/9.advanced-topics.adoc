=== Schema changes

Schema changes are handled online provided an initial consistent schema image was created.
Adding a new schema to the replication list requires creating or restoring its schema snapshot and possibly restarting the process.

[CAUTION]
====
Tracking many schemas may impact performance.
====

Schema updates triggered by DDL are applied automatically when the parser processes the system-table DML.

=== Configuration updates

The program reads `OpenLogReplicator.json` on startup and monitors it for changes to table-level filters.
You can modify table and owner filters online; the program reloads the configuration when the file modification timestamp changes.

[CAUTION]
====
Configuration reloads are not atomic.
Long-running transactions may span old and new configurations, producing outputs that mix both settings.
====

image:buffers-in-memory1.png[Memory buffers before configuration change,,,]
image:buffers-in-memory2.png[Memory buffers after configuration change,,,]

[IMPORTANT]
====
Only `table` and `owner` entries in `debug` and `filter` are hot-reloaded.
Other parameters require a restart.
====

=== Offline reader

Offline mode (`"type": "offline"`) reads redo files without a database connection.
A schema file created with `gencfg.sql` or exported from a prior online run is required.

[CAUTION]
====
Offline schema files include static redo file locations.
If paths change, edit the JSON schema file or recreate it.
====

==== Creating an offline schema

Run `gencfg.sql` (in `scripts`) against a suitable database copy.
Provide `v_NAME`, `v_USERNAME_LIST`, and `v_SCN`; save the output to `checkpoint/<name>-chkpt-<scn>.json`, and create `checkpoint/<name>-chkpt.json` containing `{"database":"<name>","scn":<scn>,"resetlogs":<resetlogs>,"activation":<activation>}`.

=== DataGuard

Physical DataGuard is supported.
Start the initial run on the primary; thereafter the program detects role changes and follows the active redo stream automatically.

=== High availability

Network mode enables HA patterns where the target confirms processed SCNs.
Store checkpoints in durable external storage (future releases may add Redis).
Stream mode is recommended for HA.

=== Schemaless mode

Schemaless mode (`"flags": 2`) decodes redo data without schema information.
Values are emitted as hex strings and column/table names are unavailable.

[IMPORTANT]
====
Schemaless mode cannot filter by table and outputs all tables.
It is intended for debugging and analysis, not production.
====

Example schemaless output:
"payload":[{"op":"c","schema":{"table":"OBJ_87705"},"after":{"COL_0":"53544f50","COL_1":"787708010d2339"}}]

[TIP]
====
Use schemaless mode when only redo files are available.
Combine with adaptive schema for better analysis.
====

=== Adaptive schema mode

Adaptive schema (`"flags": 4`) tries to operate using a default base schema when a schema file is missing.
The default schema file is `base-<db-version>.json` and must be created manually by running OpenLogReplicator on an empty/representative database and copying the checkpoint schema file.

[CAUTION]
====
`base-*.json` is not distributed with the product.
Create it by running the program and exporting the generated schema snapshot.
====

OpenLogReplicator would act in hybrid mode:

- when a new schema is created: the schema would be extended by the new objects -- for such tables, the schema information would be present in the output;
- for existing tables: the schema information would be missing in the output -- output would be in schemaless mode.

[IMPORTANT]
====
The adaptive schema mode is not intended for production use.
It is intended for debugging and analysis of the data in the redo log.
====

[CAUTION]
====
This mode is less restrictive when it comes to schema changes.
In cases when OpenLogReplicator would normally stop because of schema change, it would continue in the adaptive schema mode.
====
