OpenLogReplicator supports multiple writer targets: Kafka, file, network (plain TCP), and ZeroMQ.

=== Kafka target

Kafka writer uses librdkafka (C/C++) and is optional at build time.

==== Build

Enable with the build argument `WITHKAFKA=1` (Docker) or by enabling the Kafka module during build.
The Kafka client is native C/C++ — no Java runtime required.

[CAUTION]
====
librdkafka enforces a maximum message size ≈ 953 MB.
OpenLogReplicator can process larger logical messages (multi-LOB rows) and may split or warn when messages exceed broker/client limits.
====

==== Idempotence

Idempotent producer is enabled by default via producer properties.
Kafka transactions (the transactional API) are not supported.

==== Performance

Messages are sent asynchronously and the writer does not block on broker acknowledgments.
Configure `max-messages` and other producer properties to tune throughput.

==== Checkpointing

Kafka target checkpointing follows the same scheme as the file target.

=== File target [[file-target]]

File target writes messages to disk and has no inherent size limits.
Use the file target for reproducible debugging or offline analysis.

==== Checkpointing

Checkpoints are stored in files in the checkpoint directory by default.
Two types:

- SCN checkpoint — records the last processed SCN.
- Schema snapshots — periodic files that store the schema image at a specific SCN.

[CAUTION]
====
Schema snapshots are created only when the schema changed since the last snapshot.
The program deletes old snapshots by default; options control retention.
====

=== Network target

Network writer streams data over TCP or ZeroMQ to a single client.
The client controls checkpoint confirmation and thus the Low Watermark.

- Supported protocols: `network` (plain TCP) and `zeromq`.
- Only one client may connect at a time (current implementation).
- Communication is bidirectional; the client may request INFO/START/REDO.

[IMPORTANT]
====
The receiver is responsible for persisting the last confirmed SCN.
On reconnect, the receiver should supply the last confirmed SCN so the sender can resume.
====

[IMPORTANT]
====
Once the Low Watermark advances and transactions are removed from memory, they cannot be retried by the sender.
====
