== Quick start

For a quick start, see the tutorials in the GitHub repository at {url-github-tutorials}[tutorials].

=== Running in Docker

Development and testing are performed using Docker.
Docker is the recommended environment for running and testing the program.
For Docker images and build instructions, see {url-github-docker}[GitHub].

== Architecture

OpenLogReplicator is written in C++ and uses the database client libraries to read redo logs directly.

Transactions are parsed from redo log files (online or archived).
The process requires physical access to redo files; files must not be deleted before processing.

[HINT]
====
Refer to the database documentation to configure redo retention.
====

The program is multithreaded; key threads:

- Main thread — startup and shutdown.
- Reader threads — read redo files from disk (one per redo log group).
Most readers sleep until their file becomes current.
- Checkpoint thread — writes checkpoint and schema snapshots (default: every 10 seconds).
- Parser thread — single-threaded, CPU-bound parser.
It decodes redo records, updates schema on DDL, and prepares messages for output.
- Writer thread(s) — send transactions to configured targets; may use additional threads depending on connector libraries.

[NOTE]
====
Only the parser is CPU intensive.
Other threads are primarily I/O-bound.
====

[IMPORTANT]
====
To avoid multiple instances overwriting checkpoints, the program locks the configuration file on startup.
If the lock fails because another instance runs, the program exits.
====

=== Memory allocation [[memory-allocation]]

Most runtime buffers (disk buffers, output buffers, redo vectors) are allocated from a configurable memory pool controlled by `min-mb` and `max-mb`.
Metadata (table names, column definitions, etc.) and LOBs are allocated on the heap.

[CAUTION]
====
LOB data currently uses dynamic heap allocation; a future release will move LOBs into the memory pool.
Large LOB-heavy streams can therefore exceed configured memory limits.
====

[HINT]

==== Memory swapping

For best performance, run OpenLogReplicator on a host with enough physical memory to avoid OS-level swapping.
If memory is constrained, the program can swap transaction data to disk; metadata and read buffers always remain in memory.

[IMPORTANT]
====
Only transaction data can be swapped; metadata and read buffers are retained in memory.
====

When swapping is active, the oldest inactive transaction blocks are chosen first.
For each open transaction at least the current block remains in memory.

image:memory-swapping.png[Memory swapping,,,]

==== Sizing memory

OpenLogReplicator uses very strict memory allocation and tries to not copy the redo data mulitple times in memory.

[HINT]
====
If the database transactions are small - the program can work with very low memory footprint like 32MB memory.
====

Recommended steps to size memory:

1. Set `min-mb` — minimum reserved memory for the process.
2. Set `max-mb` — maximum physical memory available for the process.
3. Set `swap-mb` — memory level at which the program starts swapping transaction data.

[IMPORTANT]
====
Set `max-mb` to at least 110% of the maximum expected single LOB size (e.g. for a 1 GB LOB set `max-mb` ≥ 1.1 GB) to avoid error 10072.
====

Performance testing is recommended.

=== Database redo logs

Database DML (INSERT/UPDATE/DELETE) is recorded in redo logs.
Redo records do not include a complete schema snapshot for every transaction; schema metadata is cached and updated on DDL.

On first run, OpenLogReplicator collects schema metadata from system tables and stores it locally for subsequent runs.
If the schema changes, the metadata is updated accordingly.

[IMPORTANT]
====
OpenLogReplicator does not perform an initial data load.
It reads redo logs only and does not execute SELECT queries against application data.
Use ETL or backup/restore tools for initial data sync.
====

[CAUTION]
====
All redo log blocks required for the active window must be available.
Missing blocks require reinitializing replication and rebuilding the schema snapshot.
====

=== Transaction processing

Redo logs include committed, in-progress, and rolled-back transactions.
OpenLogReplicator:

- Ignores rolled-back transactions or rolled-back DML segments.
- Emits committed transactions to output when the COMMIT record is processed.
- Preserves transaction ordering by commit SCN and does not interleave DML from different transactions in the output.
- May split a transaction into multiple output messages (e.g., begin/each DML/commit), depending on configuration.

[NOTE]
====
DMLs from different transactions are interleaved in redo logs; the parser reconstructs proper transaction order.
====

image:transaction-interleaving.png[Transform interleaved transactions to stream,,,]

=== Transaction caching and restart

Open transactions are cached in memory until commit.
On restart, the program resumes from the Low Watermark (the oldest unprocessed transaction), which may require reprocessing earlier redo data.
Checkpoints do not contain full transaction bodies, so restarts can be time- and resource-consuming.

[CAUTION]
====
Restarting may require reprocessing old redo files.
Configure redo retention to retain enough history to recover.
====

[TIP]
====
Avoid unnecessary restarts.
Keep sufficient redo retention for recovery.
====

.Replication start example
image:replication-start.png[Replication start example,,,]

[NOTE]
====
In the example above, _Transaction 2_ and _Transaction 4_ have already been processed and would not be processed again.
Since OpenLogReplicator doesn't cache in the checkpoint files transaction DML commands, all redo log data need which would contain it has to be processed again after restart.
This would include data for _Transaction 1_ and _Transaction 3_.
====

On startup, OpenLogReplicator reads from the beginning of some redo log file.
It is up to the user to decide the moment from which the redo log is parsed.
There could be always some transactions that are not yet committed, but started earlier.

[CAUTION]
====
On startup, all transactions that started the moment ago of startup are discarded.
====

=== Topology

Two deployment options:

==== Local on database host

Simplest and fastest I/O, but may impact database performance under resource contention.

[CAUTION]
====
Ensure sufficient CPU and memory remain available for the database.
====

==== Remote host

Recommended for production.
Ensure redo files are readable remotely by using:

- read-only remote mounts (SSHFS),
- standby database copies,
- SRDF or other replication,
- archived redo logs copied by batch process.

== Output format [[output-format]]

Two formats are supported: JSON and Protocol Buffers (experimental).

=== JSON format

JSON is fast and typically smaller in generation overhead.
The writer constructs messages directly during parsing to minimize allocations.

==== Response fields

- scn/scns — SCN value (decimal in `scn` or hex string in `scns`).
See xref:../json/format.adoc#scn[scn].
- tm/tms — timestamp (numeric or string).
See xref:../json/format.adoc#timestamp[timestamp].
- xid/xidn — transaction ID formats.
See xref:../json/format.adoc#xid[xid].
- db — database name.
See xref:../json/format.adoc#db[db].
- payload.op — operation type: `"begin"`, `"commit"`, `"c"`, `"u"`, `"d"`, `"ddl"`, `"chkpt"`.
- payload.schema — included in DML messages; contains `owner`, `table`, `obj`, and `columns`.
- payload.before / payload.after — column value arrays for UPDATE/DELETE and INSERT/UPDATE respectively.
Only values present in redo are emitted; unchanged columns may be omitted.
- payload.rid — row ID when available.
- DDL messages may include the SQL text and parsed system-table DML used to update the internal schema.

[CAUTION]
====
Only data available in the redo log is emitted.
For updates, unchanged columns may be absent.
====

=== Protocol Buffers

Protobuf uses the same logical field names and types as JSON, but constructs messages field-by-field before serialization.
Memory and CPU overhead are higher than JSON.

[IMPORANT]
====
The status of Protocol Buffers format is in Experimental state.
It be unstable and not fully implemented.
====


== Output targets

OpenLogReplicator supports multiple writer targets: Kafka, file, network (plain TCP), and ZeroMQ.

=== Kafka target

Kafka writer uses librdkafka (C/C++) and is optional at build time.

==== Build

Enable with the build argument `WITHKAFKA=1` (Docker) or by enabling the Kafka module during build.
The Kafka client is native C/C++ — no Java runtime required.

[CAUTION]
====
librdkafka enforces a maximum message size ≈ 953 MB.
OpenLogReplicator can process larger logical messages (multi-LOB rows) and may split or warn when messages exceed broker/client limits.
====

==== Idempotence

Idempotent producer is enabled by default via producer properties.
Kafka transactions (the transactional API) are not supported.

==== Performance

Messages are sent asynchronously and the writer does not block on broker acknowledgments.
Configure `max-messages` and other producer properties to tune throughput.

==== Checkpointing

Kafka target checkpointing follows the same scheme as the file target.

=== File target [[file-target]]

File target writes messages to disk and has no inherent size limits.
Use the file target for reproducible debugging or offline analysis.

==== Checkpointing

Checkpoints are stored in files in the checkpoint directory by default.
Two types:

- SCN checkpoint — records the last processed SCN.
- Schema snapshots — periodic files that store the schema image at a specific SCN.

[CAUTION]
====
Schema snapshots are created only when the schema changed since the last snapshot.
The program deletes old snapshots by default; options control retention.
====

=== Network target

Network writer streams data over TCP or ZeroMQ to a single client.
The client controls checkpoint confirmation and thus the Low Watermark.

- Supported protocols: `network` (plain TCP) and `zeromq`.
- Only one client may connect at a time (current implementation).
- Communication is bidirectional; the client may request INFO/START/REDO.

[IMPORTANT]
====
The receiver is responsible for persisting the last confirmed SCN.
On reconnect, the receiver should supply the last confirmed SCN so the sender can resume.
====

[IMPORTANT]
====
Once the Low Watermark advances and transactions are removed from memory, they cannot be retried by the sender.
====

image:network-target-architecture.png[Network target architecture,,,]

== Supported features

=== DML operations

Supports INSERT (including multi-row INSERT), UPDATE, and DELETE (including multi-row DELETE).
Direct-path INSERT (/*+ append */) is not supported.

=== Table characteristics

Supports nullable columns, invisible columns, default values, up to 1000 columns (till 21c) or 4095 colums (23ai), row chaining, savepoint rollbacks, partitioned tables, row dependencies, and BASICFILE LOBs.

Transactions that are rolled back are ignored.

=== Column types

Supported types include (examples): varchar2, number, date, raw, xmltype (CLOB storage), char, binary_float/double, CLOB, BLOB, timestamps, intervals, urowid, timestamp with local time zone, boolean.
Unsupported types are emitted as `"?"`.

=== DDL and schema updates

DDL is processed by parsing associated system-table DML and updating the internal schema.
DDL may span multiple transactions (e.g., table then indexes).
Temporary inconsistencies (e.g., missing index) may cause transient warnings; these can be ignored.

=== Supported character sets [[supported-character-sets]]

All source character fields are read according to the source character set.
Output is always UTF-8 (Unicode).
Character-set conversion can be disabled for testing.
See xref:../json/format.adoc#char[char].

A broad list of supported character sets is implemented; the target is always UTF-8.

== Replication

On first run, a schema snapshot for tracked schemas is created.
Only configured schemas are collected by default.

If you add schemas to the configuration after the first run, replication will not start for them until schema snapshots exist.
Recreating the schema (or removing checkpoints) may be required.

=== Schema creation

Schemas and tables to replicate are defined in the configuration; regex wildcards are supported.

If a schema does not yet exist at startup, OpenLogReplicator begins tracking it once it appears.

== Startup

=== Schema

On startup the program loads the schema snapshot from the checkpoint store.
The schema is a consistent copy of system tables at a single SCN (`AS OF SCN` queries).
If the DB UNDO/retention is insufficient, generate the schema using `gencfg.sql` on a suitable database copy.

=== Checkpoints

On first run checkpoints are created in the configured checkpoint directory.
Each schema snapshot is associated with the SCN at which it was taken.

=== Start position

Replication can start by SCN, by time (converted with `TIMESTAMP_TO_SCN`), or by specific redo file and offset.
Avoid starting too far in the past — a consistent schema snapshot may be unavailable.

== Checkpointing

OpenLogReplicator advances positions only after whole LWNs (Log Writer Number blocks) are processed.
After each LWN a checkpoint record is emitted; by default checkpoint records are enabled.

Example checkpoint payload:
"payload":[{"op":"chkpt","seq":1763,"offset":15872}]

To disable checkpoint messages set the reader `flags` accordingly.

Checkpoint records include `seq` (redo file sequence), `offset` (byte offset), and optionally `"redo":true` when a redo switch occurred.
Checkpoints are useful as heartbeats — they are emitted every few seconds in a healthy stream.

== Advanced topics

=== Schema changes

Schema changes are handled online provided an initial consistent schema image was created.
Adding a new schema to the replication list requires creating or restoring its schema snapshot and possibly restarting the process.

[CAUTION]
====
Tracking many schemas may impact performance.
====

Schema updates triggered by DDL are applied automatically when the parser processes the system-table DML.

=== Configuration updates

The program reads `OpenLogReplicator.json` on startup and monitors it for changes to table-level filters.
You can modify table and owner filters online; the program reloads the configuration when the file modification timestamp changes.

[CAUTION]
====
Configuration reloads are not atomic.
Long-running transactions may span old and new configurations, producing outputs that mix both settings.
====

image:buffers-in-memory1.png[Memory buffers before configuration change,,,]
image:buffers-in-memory2.png[Memory buffers after configuration change,,,]

[IMPORTANT]
====
Only `table` and `owner` entries in `debug` and `filter` are hot-reloaded.
Other parameters require a restart.
====

=== Offline reader

Offline mode (`"type": "offline"`) reads redo files without a database connection.
A schema file created with `gencfg.sql` or exported from a prior online run is required.

[CAUTION]
====
Offline schema files include static redo file locations.
If paths change, edit the JSON schema file or recreate it.
====

==== Creating an offline schema

Run `gencfg.sql` (in `scripts`) against a suitable database copy.
Provide `v_NAME`, `v_USERNAME_LIST`, and `v_SCN`; save the output to `checkpoint/<name>-chkpt-<scn>.json`, and create `checkpoint/<name>-chkpt.json` containing `{"database":"<name>","scn":<scn>,"resetlogs":<resetlogs>,"activation":<activation>}`.

=== DataGuard

Physical DataGuard is supported.
Start the initial run on the primary; thereafter the program detects role changes and follows the active redo stream automatically.

=== High availability

Network mode enables HA patterns where the target confirms processed SCNs.
Store checkpoints in durable external storage (future releases may add Redis).
Stream mode is recommended for HA.

=== Schemaless mode

Schemaless mode (`"flags": 2`) decodes redo data without schema information.
Values are emitted as hex strings and column/table names are unavailable.

[IMPORTANT]
====
Schemaless mode cannot filter by table and outputs all tables.
It is intended for debugging and analysis, not production.
====

Example schemaless output:
"payload":[{"op":"c","schema":{"table":"OBJ_87705"},"after":{"COL_0":"53544f50","COL_1":"787708010d2339"}}]

[TIP]
====
Use schemaless mode when only redo files are available.
Combine with adaptive schema for better analysis.
====

=== Adaptive schema mode

Adaptive schema (`"flags": 4`) tries to operate using a default base schema when a schema file is missing.
The default schema file is `base-<db-version>.json` and must be created manually by running OpenLogReplicator on an empty/representative database and copying the checkpoint schema file.

[CAUTION]
====
`base-*.json` is not distributed with the product.
Create it by running the program and exporting the generated schema snapshot.
====

OpenLogReplicator would act in hybrid mode:

- when a new schema is created: the schema would be extended by the new objects -- for such tables, the schema information would be present in the output;
- for existing tables: the schema information would be missing in the output -- output would be in schemaless mode.

[IMPORTANT]
====
The adaptive schema mode is not intended for production use.
It is intended for debugging and analysis of the data in the redo log.
====

[CAUTION]
====
This mode is less restrictive when it comes to schema changes.
In cases when OpenLogReplicator would normally stop because of schema change, it would continue in the adaptive schema mode.
====
